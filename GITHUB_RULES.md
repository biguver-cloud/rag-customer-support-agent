# GitHub運用ルール

## ブランチ戦略

| ブランチ名 | 役割 |
|-----------|------|
| `main` | 本番環境相当。直接pushは禁止 |
| `develop` | 開発の統合ブランチ |
| `feature/#課題番号-機能名` | 機能追加（例: `feature/#12-user-auth`） |
| `fix/#課題番号-修正内容` | バグ修正（例: `fix/#15-login-error`） |

---

## コミットメッセージ規則

[Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/) に準拠する。

| プレフィックス | 用途 |
|-------------|------|
| `feat:` | 新機能の追加 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントの変更 |
| `refactor:` | 機能変更を伴わないコード改善 |
| `test:` | テストの追加・修正 |
| `chore:` | ライブラリ更新・設定変更など |

**例**

```
feat: ログイン機能を追加
fix: パスワードバリデーションのバグを修正
docs: READMEにセットアップ手順を追記
```

---

## プルリクエスト運用

- 1PR = 1機能 or 1修正（小さく保つ）
- 以下の内容を記載してマージリクエストを作成する
  - 変更内容の概要
  - 動作確認方法
  - スクリーンショット（UI変更がある場合）
- マージ前にセルフレビューを実施する
- `main` への直接マージは禁止。必ず `develop` を経由する

---

## Issue管理

- 機能追加・バグ修正はIssueを作成してから着手する
- 以下のラベルで分類する

| ラベル | 用途 |
|-------|------|
| `enhancement` | 機能追加 |
| `bug` | バグ報告・修正 |
| `docs` | ドキュメント関連 |

- ブランチ名にはIssue番号を含める（例: `feature/#12-user-auth`）

---

## マージ戦略

| マージ先 | 方式 | 理由 |
|---------|------|------|
| `develop` | Squash merge | コミット履歴をまとめてログを見やすく保つ |
| `main` | Merge commit | リリース履歴を明示的に残す |
