# GitHub運用ルール

## 背景・目的

以下の課題を解消するため、本リポジトリではGitHub運用ルールを定めています。

- コミットメッセージやIssueの記述が不十分で、変更内容が不透明になりやすい
- テンプレートが統一されておらず、メンバー間で認識の齟齬が生じる
- ブランチ名や運用が統一されず、mainへの誤pushが発生しやすい

---

## ブランチ戦略

ポートフォリオ単位のためシンプルに管理しやすい**GitHub Flowを採用**。

| ブランチ名 | 役割 |
|-----------|------|
| `main` | 常に最新の状態を保持。直接commitは禁止、PRのみでマージ |
| `feature/name` | 新しい機能を追加するブランチ。mainからの派生。完了後PRでmainにマージ |
| `hotfix/name` | PR後のバグ修正専用ブランチ。mainからの派生。完了後PRでmainにマージ |

### リリースまでのフロー

```
1. mainブランチからfeature/name または hotfix/name ブランチを切る
2. 各ブランチで開発
3. PR作成
4. セルフレビュー実施（実務ではチームメンバーによる承認を前提とする）
5. 承認後mainブランチへマージ
6. マージ完了後、該当ブランチを削除する
```

---

## Issue管理

### ルール

- 機能追加・バグ修正はIssueを作成してから着手する
- チーム全体で問題内容を共有するため、作業前にIssueで課題を明確化する
- Issueを作成したら必ず自分をAssigneeに設定する
- ブランチ名にはIssue番号を含める

### ラベル一覧

| ラベル | 用途 |
|-------|------|
| `enhancement` | 機能追加 |
| `bug` | バグ報告・修正 |
| `docs` | ドキュメント関連 |

### ブランチ命名規則

```
feature/#12-ログイン機能
hotfix/#13-パスワードバグ修正
```

### IssueからマージまでのFlow

```
1. Issueを作成し、ラベルを付与・Assigneeに自分を設定
2. Issue番号を含めたブランチを切る
3. 開発・PR作成
4. マージ時にclose #Issue番号を記載し、Issueを自動クローズ
```

### Issueテンプレート

```
## 何が問題か
（例: ログイン後にページが遷移しない）

## 問題を解消する目的
（例: 未認証ユーザーがマイページにアクセスできてしまうため）

## 再現手順・スクリーンショット
（例: ①ログインボタンを押す ②画面が変わらない）

## 対応方針
（例: 認証後のリダイレクト処理を修正する）
```

## Pull Request Convention

### ルール

- **レビュー可能な単位**：10〜20ファイル以内を推奨
- **1つのテーマ**：PRは1つのテーマに絞る。バグ修正と新機能追加を1つのPRに含めない
  - ✅ Aを実装した → PR：A
  - ❌ AとBを実装した → PR：A・Bをまとめて作成
- **必ずレビューを受けてからマージ**：個人開発のためセルフレビューを実施（実務ではチームメンバーによる承認を前提とする）
- **承認後の変更は再承認**：承認後にPR内容の変更があった場合は再度承認を受ける
- **タイトルと説明の明記**：変更内容と目的を明確に記載する（テンプレートに準拠）

### セルフレビューチェックリスト

- [ ] 動作確認済みか
- [ ] 不要なコメントアウトが残っていないか
- [ ] console.logなどのデバッグコードが残っていないか

### PRテンプレート

```
## 関連Issue
close #

## 行った作業
（例: ログイン機能を追加した）

## 作業の目的
（例: 未認証ユーザーのアクセスを制限するため）

## before → after
Before:
After:
（スクリーンショットまたはデモ動画を添付）

## 結論・確認事項
- [ ] ○○の動作確認済み
- [ ] ○○については未対応（理由：）
```

## Commit Convention

### ルール

- **1コミット = 1つの変更**：複合すると何がコミットされたか不明になるため
- **小さなコミット**：大きな変更は小さなコミットに分ける
- **意味のあるメッセージ**：何をしたか一目でわかるように記載する
- **基本日本語で作成**
- **Prefixをつける**：変更の目的をカテゴリで宣言する

### Prefix一覧

| Prefix | 用途 |
|--------|------|
| `feat:` | 新しい機能の追加 |
| `fix:` | バグの修正 |
| `docs:` | ドキュメントのみの変更 |
| `style:` | 空白・フォーマット・セミコロン追加など |
| `refactor:` | 仕様に影響がないコード改善 |
| `perf:` | パフォーマンス向上関連 |
| `test:` | テスト関連 |
| `chore:` | ビルド・補助ツール・ライブラリ関連 |

### 記載例

```
feat: ログイン機能を追加
fix: パスワードバリデーションのバグを修正
docs: READMEにセットアップ手順を追記
refactor: 認証処理をサービス層に分離
```
